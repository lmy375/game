# 战棋类游戏 PRD v0.1

## 1. 产品概述

### 1.1 项目代号

暂定名：**Project Formation**

中文暂定名：**阵形之术**

### 1.2 游戏类型

回合制战棋策略游戏。

核心玩法聚焦于：

**横版/棋盘格站位、AOE 技能形状、位移控制、阵型塑造、连招收益最大化。**

玩家通过移动角色、释放位移技能、改变敌我双方站位，再使用不同形状的 AOE 技能打出高收益伤害或控制效果。

### 1.3 目标平台

第一阶段目标平台：

* Web 浏览器
* 微信小游戏

第一阶段不考虑：

* Steam
* iOS / Android 原生 App
* 主机平台

### 1.4 推荐技术栈

客户端：

* PixiJS
* TypeScript
* 2D / 2.5D 棋盘表现
* Web + 微信小游戏双端发布

架构原则：

* 战斗核心逻辑使用纯 TypeScript 实现
* PixiJS 只负责表现层、输入、动画、音效、UI
* 核心逻辑不依赖具体渲染引擎的节点、生命周期和渲染 API
* 所有技能、角色、关卡、地形尽量数据驱动

推荐代码结构：

```text
src/
  game-core/
    board/
    unit/
    turn/
    skill/
    pattern/
    effect/
    displacement/
    simulator/
    evaluator/
    ai/
  game-data/
    units.json
    skills.json
    patterns.json
    levels.json
  platform/
    pixi/
    wechat/
```

## 2. 产品定位

### 2.1 一句话定位

一款以“改变站位制造 AOE 最大收益”为核心乐趣的回合制战棋游戏。

### 2.2 核心卖点

本游戏不是单纯比较数值的战棋，而是强调：

1. 通过技能改变敌我站位
2. 通过 AOE 形状匹配敌人阵型
3. 通过连招制造高收益回合
4. 通过地图地形放大位移与站位价值
5. 通过强预览降低操作误判

### 2.3 设计目标

玩家每一回合的核心思考应是：

```text
当前敌人站位是什么？
我能否把敌人推成一排？
我能否把敌人聚到十字中心？
我能否用位移技能把敌人送入陷阱？
我是否应该先改变站位，再释放 AOE？
我这一回合能否形成连招？
```

最终形成的核心循环：

```text
观察站位 → 改造站位 → 释放 AOE → 处理残局 → 防止敌人反制
```

## 3. 目标用户

### 3.1 核心用户

喜欢以下游戏类型的玩家：

* 回合制战棋
* 策略 RPG
* 站位解谜
* 技能连招
* 小队养成
* 关卡挑战

参考用户偏好：

* 喜欢《三角战略》的战棋节奏
* 喜欢《Into the Breach》的空间解谜感
* 喜欢《火焰纹章》的角色与职业感
* 喜欢《神界：原罪 2》中的地形和技能组合感

### 3.2 目标体验

玩家应获得的主要爽点：

* “我把 3 个敌人推成一排，然后一枪穿了 3 个”
* “我先用风把敌人聚到中心，再用十字火焰打满伤害”
* “我把敌人推进陷阱，再用 AOE 收割”
* “我预判敌人站位，提前用障碍物卡住他们”
* “这一回合不是靠数值赢，而是靠站位设计赢”

## 4. 核心玩法概述

### 4.1 战斗形态

游戏采用回合制战棋战斗。

基础元素包括：

* 棋盘格
* 我方单位
* 敌方单位
* 地形
* 障碍物
* 技能
* AOE 范围
* 位移效果
* 状态效果
* 胜负条件

### 4.2 棋盘结构

第一阶段采用方格棋盘。

每个格子拥有以下属性：

```text
坐标
地形类型
是否可移动
是否可站立
是否阻挡视线
是否阻挡技能
是否阻挡位移
高度
特殊效果
当前占用单位
```

基础地形类型：

| 地形   | 效果               |
| ---- | ---------------- |
| 普通地面 | 可移动、可站立          |
| 墙体   | 阻挡移动、阻挡位移、阻挡部分技能 |
| 障碍物  | 阻挡移动，可被部分技能破坏    |
| 水面   | 可触发导电、减速等效果      |
| 火焰地形 | 进入或停留时受到伤害       |
| 陷阱格  | 单位进入时触发效果        |
| 高台   | 影响射程和部分技能效果      |
| 悬崖边缘 | 被击退时可能坠落或受到额外伤害  |

MVP 阶段只实现：

* 普通地面
* 墙体
* 障碍物
* 火焰地形
* 陷阱格

## 5. 战斗核心循环

### 5.1 回合流程

MVP 阶段采用我方整队回合、敌方整队回合。

流程：

```text
战斗开始
↓
我方回合开始
↓
选择单位
↓
移动 / 技能 / 待机
↓
所有我方单位行动结束
↓
敌方回合开始
↓
敌人 AI 行动
↓
回合结束
↓
判断胜负
```

后续版本可扩展为基于速度的行动顺序系统。

### 5.2 单位行动

每个单位每回合默认拥有：

* 1 次移动
* 1 次主动技能
* 可选择待机

MVP 阶段暂不实现复杂行动点系统。

后续可扩展：

* AP 行动点
* 技能冷却
* 能量值
* 连携点
* 反应技

## 6. AOE 技能系统

### 6.1 设计原则

AOE 是本游戏的核心系统。

AOE 技能不能只表示“半径范围”，而应拆分为：

```text
施法范围 Cast Range
命中形状 Impact Pattern
格子效果 Cell Effect
方向 Direction
命中规则 Hit Rule
结算顺序 Resolve Order
```

### 6.2 施法范围

施法范围决定玩家能选择哪些目标点、目标单位或方向。

常见施法范围：

| 类型     | 描述              |
| ------ | --------------- |
| 单点目标   | 选择一个单位或一个格子     |
| 距离范围   | 选择距离施法者 N 格内的格子 |
| 直线目标   | 选择施法者前方直线方向     |
| 自身周围   | 以施法者自身为中心       |
| 指定方向   | 上、下、左、右四方向      |
| 目标单位周围 | 以目标单位为中心生成范围    |

### 6.3 命中形状 Pattern

命中形状使用相对坐标描述。

示例：十字型。

```json
{
  "id": "cross_1",
  "cells": [
    { "dx": 0, "dy": 0, "effectKey": "center" },
    { "dx": 1, "dy": 0, "effectKey": "arm" },
    { "dx": -1, "dy": 0, "effectKey": "arm" },
    { "dx": 0, "dy": 1, "effectKey": "arm" },
    { "dx": 0, "dy": -1, "effectKey": "arm" }
  ]
}
```

示例：直线型。

```json
{
  "id": "line_4",
  "anchor": "caster_direction",
  "rotatable": true,
  "cells": [
    { "dx": 0, "dy": 1, "effectKey": "near" },
    { "dx": 0, "dy": 2, "effectKey": "mid" },
    { "dx": 0, "dy": 3, "effectKey": "mid" },
    { "dx": 0, "dy": 4, "effectKey": "far" }
  ]
}
```

示例：T 字型。

```json
{
  "id": "t_shape",
  "anchor": "caster_direction",
  "rotatable": true,
  "cells": [
    { "dx": 0, "dy": 1, "effectKey": "stem" },
    { "dx": -1, "dy": 2, "effectKey": "head" },
    { "dx": 0, "dy": 2, "effectKey": "head" },
    { "dx": 1, "dy": 2, "effectKey": "head" }
  ]
}
```

### 6.4 Pattern 类型

MVP 阶段需要支持以下 Pattern：

| Pattern | 用途           |
| ------- | ------------ |
| 单点      | 普通攻击、治疗      |
| 十字型     | 聚怪后爆发        |
| 直线型     | 奖励敌人排成一列     |
| T 字型    | 奖励敌人处于前排横向结构 |
| 3x3 区域  | 基础范围控制       |
| 扇形，后续版本 | 侧翼攻击         |
| 环形，后续版本 | 包围战术         |

### 6.5 Pattern 旋转

所有带方向的 Pattern 必须支持旋转。

初始 Pattern 默认朝上。

系统根据释放方向将相对坐标旋转到最终坐标。

支持方向：

```text
上
下
左
右
```

后续可扩展：

```text
左上
右上
左下
右下
```

### 6.6 Cell Effect

同一个 Pattern 中，不同格子可以有不同效果。

示例：十字火焰。

```json
{
  "id": "cross_fire",
  "pattern": "cross_1",
  "cellEffects": {
    "center": [
      { "type": "damage", "element": "fire", "multiplier": 1.5 },
      { "type": "apply_status", "status": "burn" }
    ],
    "arm": [
      { "type": "damage", "element": "fire", "multiplier": 0.8 }
    ]
  }
}
```

设计目标：

* 中心格可以高伤害
* 边缘格可以低伤害
* 某些格子可以只控制不伤害
* 某些格子可以附加状态
* 某些格子可以触发位移

## 7. 位移系统

### 7.1 设计原则

位移系统是本游戏的第二核心系统。

位移能力不是普通附加效果，而是改变战斗格局的关键手段。

玩家应通过位移技能实现：

```text
聚怪
排线
推入陷阱
推入火焰
推下高台
打散敌人
保护队友
破坏敌方阵型
制造 AOE 最大收益
```

### 7.2 位移类型

MVP 阶段需要支持：

| 类型        | 描述          |
| --------- | ----------- |
| Push      | 将目标沿方向推开    |
| Pull      | 将目标向中心拉近    |
| Gather    | 将多个目标聚向中心   |
| Swap      | 交换两个单位位置    |
| LineUp    | 将多个单位整理成直线  |
| Knockback | 击退，撞墙造成额外伤害 |

后续扩展：

| 类型             | 描述        |
| -------------- | --------- |
| Scatter        | 打散单位      |
| Rotate         | 围绕中心旋转    |
| Drag           | 拖拽到指定格    |
| Teleport       | 传送        |
| SummonObstacle | 召唤障碍物改变站位 |

### 7.3 位移规则

位移必须可预测。

基础规则：

1. 单位不能被移动到不可站立格
2. 单位不能被移动到已有单位占据的格子
3. 位移遇到墙体或障碍物时停止
4. 被推撞墙时可触发撞击伤害
5. 被推入火焰或陷阱格时触发地形效果
6. 多个单位向同一格移动时，根据优先级依次结算
7. 飞行单位后续可拥有特殊位移规则，MVP 暂不实现

### 7.4 聚怪技能示例

技能名：狂风聚拢

```json
{
  "id": "gale_gather",
  "name": "狂风聚拢",
  "castRange": {
    "type": "distance",
    "min": 1,
    "max": 4
  },
  "pattern": "cross_2",
  "effects": {
    "all": [
      {
        "type": "pull_to_center",
        "maxDistance": 2,
        "collision": "stop_before_occupied"
      }
    ]
  }
}
```

设计效果：

* 选择一个中心点
* 影响中心附近敌人
* 将敌人向中心点拉近
* 不造成伤害
* 用于为后续 AOE 创造阵型

### 7.5 排线技能示例

技能名：整列风压

```json
{
  "id": "line_up_wind",
  "name": "整列风压",
  "castRange": {
    "type": "distance",
    "min": 1,
    "max": 3
  },
  "pattern": "area_3x3",
  "effects": {
    "all": [
      {
        "type": "arrange_line",
        "direction": "caster_direction",
        "maxUnits": 3,
        "priority": "nearest_first"
      }
    ]
  }
}
```

设计效果：

* 将范围内最多 3 个敌人整理成直线
* 为直线型技能创造收益
* 需要强预览，展示整理后的最终站位

## 8. 技能系统

### 8.1 技能结构

技能应由以下部分组成：

```text
技能 ID
技能名称
技能描述
施法者限制
施法范围
命中形状
消耗
冷却
目标过滤
格子效果
位移效果
状态效果
动画表现
音效
AI 评分权重
```

### 8.2 技能类型

MVP 阶段技能分为：

| 类型     | 描述                |
| ------ | ----------------- |
| 普通攻击   | 单点伤害              |
| AOE 伤害 | 根据 Pattern 命中多个格子 |
| 位移控制   | 改变单位位置            |
| 状态控制   | 眩晕、易伤、燃烧等         |
| 地形交互   | 推入陷阱、点燃地面等        |
| 治疗/保护  | MVP 可选            |

### 8.3 技能结算顺序

推荐默认结算顺序：

```text
1. 检查施法合法性
2. 解析施法目标
3. 解析 Pattern 命中格
4. 过滤目标
5. 计算位移预期
6. 结算位移
7. 触发地形效果
8. 结算伤害
9. 结算状态
10. 检查死亡
11. 触发被动/后续事件
12. 更新战斗状态
```

MVP 可简化为：

```text
Pattern → 位移 → 伤害 → 状态 → 死亡判断
```

### 8.4 MVP 技能列表

MVP 需要实现 6 个核心技能。

#### 技能 1：普通攻击

用途：基础单体伤害。

```text
目标：单体敌人
范围：1 格
效果：100% 物理伤害
```

#### 技能 2：十字火焰

用途：验证聚怪后的 AOE 爆发。

```text
施法范围：1-4 格
命中形状：十字型
中心格：150% 火焰伤害 + 燃烧
四周格：80% 火焰伤害
```

#### 技能 3：贯穿射击

用途：验证直线阵型收益。

```text
施法范围：指定方向
命中形状：直线 4 格
效果：命中路径上的所有敌人
伤害：100% 物理伤害
可选规则：每多命中 1 个敌人，总收益提高
```

#### 技能 4：狂风聚拢

用途：将敌人拉向中心，为十字火焰创造收益。

```text
施法范围：1-4 格
命中形状：3x3 或十字 2
效果：敌人向中心拉近最多 2 格
伤害：无
```

#### 技能 5：横向推击

用途：把敌人推成直线，或推入陷阱/火焰。

```text
施法范围：前方 1 格
命中形状：横向 3 格
效果：将命中单位沿指定方向推 1-2 格
撞墙：造成额外伤害
```

#### 技能 6：换位术

用途：调整我方或敌方位置。

```text
施法范围：1-3 格
目标：一个单位
效果：与施法者或指定友方单位交换位置
限制：不能交换到不可站立格
```

## 9. 战斗预览系统

### 9.1 设计原则

由于本游戏强调位移和 AOE，预览系统是核心体验，不是辅助功能。

玩家在释放技能前必须看到：

```text
命中哪些格子
命中哪些单位
每个单位受到多少伤害
每个单位会被移动到哪里
是否会撞墙
是否会触发陷阱或地形
是否会死亡
释放后敌人最终站位
可能形成的后续连招机会
```

### 9.2 预览内容

技能 Hover 时：

* 显示施法范围
* 显示可选目标点
* 显示技能 Pattern

选择目标点后：

* 高亮命中格子
* 显示不同 effectKey 对应颜色或图标
* 显示命中单位
* 显示预计伤害
* 显示预计位移路径
* 显示最终落点
* 显示撞墙、陷阱、火焰等特殊提示

### 9.3 位移预览

位移技能必须展示：

```text
当前位置
移动路径
最终位置
失败原因
碰撞位置
触发地形
```

示例提示：

```text
敌人 A：被拉向中心 2 格
敌人 B：被障碍物阻挡，只移动 1 格
敌人 C：目标格被占用，无法移动
敌人 D：被推入火焰格，预计受到 20 点地形伤害
```

### 9.4 连招提示

MVP 后期可加入轻量连招提示。

例如使用“狂风聚拢”预览时，系统提示：

```text
释放后，3 名敌人将进入「十字火焰」最佳命中区域
释放后，2 名敌人将形成「贯穿射击」直线
```

第一阶段可不做智能提示，但核心模拟器需要支持后续扩展。

## 10. 敌人 AI

### 10.1 AI 设计目标

敌人 AI 需要理解基础站位风险，避免成为无脑靶子。

AI 不需要一开始非常聪明，但需要具备以下能力：

```text
攻击玩家
移动到合适距离
避免主动扎堆
避免主动排成直线
优先攻击关键位移角色
优先脱离危险区域
```

### 10.2 MVP AI

MVP 阶段采用评分函数。

每个敌人在行动时枚举可行动作：

```text
可移动位置
可释放技能
可攻击目标
```

对每个行动计算评分：

```text
行动评分 =
  伤害收益
+ 击杀收益
+ 控制收益
+ 安全收益
- 被 AOE 风险
- 站位聚集惩罚
- 排成直线惩罚
```

### 10.3 AOE 风险评估

AI 移动后需要评估：

```text
是否与队友形成直线
是否与队友聚集在十字型范围内
是否靠近火焰/陷阱
是否容易被推入危险地形
是否靠近玩家核心 AOE 角色
```

MVP 可以先实现简单规则：

* 敌人与其他敌人曼哈顿距离小于等于 1，增加风险
* 三个敌人在同一行/列且间距较近，增加风险
* 停在陷阱/火焰旁边，增加风险
* 低血量敌人优先远离 AOE 区域

## 11. 角色与职业设计

### 11.1 MVP 角色

MVP 阶段设计 3 个我方角色。

#### 角色 A：风术士

定位：位移、聚怪、阵型塑造。

技能：

* 普通攻击

## 15. 技术需求

### 15.1 核心模块

MVP 需要实现以下模块：

```text
棋盘系统
单位系统
移动范围计算
A* 寻路
技能 Pattern 解析
技能效果结算
位移模拟
战斗状态模拟器
战斗预览系统
基础敌人 AI
关卡加载
UI 操作流程
动画表现
战斗日志
胜负判断
```

### 15.2 核心逻辑要求

核心战斗逻辑必须支持：

```text
纯函数式模拟
预览不修改真实战斗状态
行动可重放
结果可测试
技能配置可扩展
AI 可调用同一套模拟器
```

推荐设计：

```text
BattleState 当前战斗状态
BattleAction 玩家/AI 的行动
BattleSimulator 模拟行动结果
BattleResult 行动结果
BattlePreview 预览结果
BattleEvaluator 评分器
```

### 15.3 模拟器接口示例

```ts
type BattleAction = {
  actorId: string;
  type: "move" | "skill" | "wait";
  moveTo?: Position;
  skillId?: string;
  targetCell?: Position;
  targetUnitId?: string;
  direction?: Direction;
};

type BattleResult = {
  nextState: BattleState;
  events: BattleEvent[];
};

interface BattleSimulator {
  simulate(state: BattleState, action: BattleAction): BattleResult;
}
```

### 15.4 事件系统

战斗结算应输出事件，而不是直接驱动表现层。

示例事件：

```ts
type BattleEvent =
  | { type: "unit_moved"; unitId: string; from: Position; to: Position }
  | { type: "unit_damaged"; unitId: string; amount: number; hpAfter: number }
  | { type: "unit_displaced"; unitId: string; from: Position; to: Position; reason: string }
  | { type: "unit_status_applied"; unitId: string; statusId: string; duration: number }
  | { type: "unit_died"; unitId: string }
  | { type: "terrain_triggered"; position: Position; terrainType: string };
```

表现层根据事件播放动画。

## 20. 风险与对策

### 20.1 风险：位移规则过于复杂

问题：

```text
玩家无法预测推拉结果
多个单位碰撞时结果不直观
```

对策：

```text
MVP 规则保持简单
强制使用清晰预览
所有位移结果在确认前展示
复杂位移后续再加
```

### 20.2 风险：AOE 形状很多但差异不明显

问题：

```text
技能只是换皮伤害
玩家感受不到策略差异
```

对策：

```text
每种形状绑定明确战术意义
直线奖励排队
十字奖励聚拢
T 字奖励前排横向结构
陷阱奖励推拉
```

### 20.3 风险：敌人 AI 太弱

问题：

```text
敌人总是扎堆，玩家无脑 AOE
```

对策：

```text
AI 加入扎堆惩罚
AI 加入直线惩罚
AI 优先攻击风术士等位移角色
AI 低血量时远离危险区域
```

### 20.4 风险：微信小游戏包体和性能限制

问题：

```text
资源过大
加载慢
低端手机卡顿
```

对策：

```text
MVP 使用轻量美术
核心逻辑纯 TS
减少大图和复杂特效
资源远程加载
优先保证战斗流畅
```

### 20.5 风险：AI 辅助开发导致代码混乱

问题：

```text
AI 一次性生成过多耦合代码
核心逻辑和表现层混在一起
```

对策：

```text
所有核心逻辑放入 game-core
每次只让 AI 实现小模块
为核心模块写单元测试
模拟器不依赖具体渲染引擎
表现层只消费 BattleEvent
```

## 22. 当前版本最重要的设计判断

本项目的第一性原理是：

**所有系统都应该服务于“站位改变带来 AOE 收益”这一核心乐趣。**

因此：

```text
新技能是否有价值，看它是否改变站位或利用站位
新职业是否有价值，看它是否制造阵型或消费阵型
新地图是否有价值，看它是否放大站位决策
新敌人是否有价值，看它是否干扰玩家阵型计划
新 UI 是否有价值，看它是否帮助玩家理解站位收益
```

如果一个系统不能服务这个目标，MVP 阶段不做。
